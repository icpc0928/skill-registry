# QA 測試框架生成器 — Prompt

> **用途**：給 AI 讀取這份文件後，自動為任何專案生成完整的 QA 測試框架。
> **使用方式**：把這份文件 + 專案資訊一起交給 AI，AI 會生成所有必要檔案。

---

## 指令

你是一位資深 QA 工程師。請根據以下提供的專案資訊，生成完整的 QA 測試框架。

### 你需要做的事

1. **分析專案** — 閱讀專案程式碼/文件，了解所有功能模組
2. **生成檔案** — 在專案根目錄建立 `qa/` 資料夾，生成以下所有檔案
3. **確保覆蓋率** — 每個功能模組都要有對應的測試案例

---

## 需要生成的檔案

```
qa/
├── README.md              # 工作流程說明
├── QA_SKILL.md           # 測試操作手冊（環境、架構、API、怎麼測）
├── TEST_CASES.md          # 測試案例總表（所有模組、所有手法）
├── reports/
│   ├── .gitkeep
│   └── TEMPLATE.md        # 測試報告模板
└── bugs/
    ├── .gitkeep
    └── TEMPLATE.md        # Bug 追蹤模板
```

## 各檔案生成規範

### 1. README.md

說明 QA 目錄結構和工作流程：

```markdown
# QA 測試目錄

## 結構
（列出 qa/ 底下所有檔案及用途）

## 工作流程

### 執行測試
1. AI 讀取 `QA_SKILL.md` 了解專案和環境
2. AI 讀取 `TEST_CASES.md` 取得所有測試案例
3. AI 逐一執行測試，記錄結果
4. AI 產出報告到 `reports/YYYY-MM-DD.md`
5. 發現 Bug 則建立 `bugs/BUG-XXX.md`

### 新增測試案例
當專案有新功能時：
1. 在 `TEST_CASES.md` 新增測試案例
2. 更新 `QA_SKILL.md`（如有架構或 API 變更）

### 回歸測試
每次部署前，執行全部 P0 + P1 測試案例，產出新報告。
```

### 3. TEST_CASES.md — 測試案例總表

#### 格式規範

```markdown
# 測試案例總表

> 編號規則：`TC-{模組縮寫}-{序號}`
>
> **測試手法標籤**：
> - 🔲 **黑盒** — 不看程式碼，只驗輸入→輸出
> - ⬜ **白盒** — 根據程式碼邏輯設計案例
> - 📏 **邊界** — 測極端值（0、-1、最大值、空值）
> - 📊 **等價類** — 把輸入分有效/無效類別
> - 🔁 **回歸** — Bug 修復後重新驗證
> - 🧭 **探索** — 不預設步驟，自由操作找問題
> - ⚡ **壓力** — 大量/高頻請求測穩定性
```

#### 每個模組必須包含

| 區塊 | 說明 |
|------|------|
| **功能測試 🔲** | 基本 CRUD、正常流程 |
| **邊界測試 📏** | 極端值：0、-1、空白、超長字串、最大值 |
| **等價類劃分 📊** | 有效/無效輸入分組（用於有輸入欄位的模組） |
| **白盒測試 ⬜** | 根據程式碼邏輯設計（用於有業務邏輯的模組） |

#### 表格格式

```markdown
| 編號 | 測試項目 | 優先級 | 類型 | 說明 |
|------|---------|--------|------|------|
| TC-XXX-001 | 測試名稱 | P0/P1/P2 | UI/API/UI+API | 詳細說明 |
```

#### 優先級定義

| 優先級 | 定義 | 範例 |
|--------|------|------|
| **P0** | 核心流程，壞了等於不能用 | 登入、下單、付款 |
| **P1** | 重要功能，影響用戶體驗 | 搜尋、篩選、管理 CRUD |
| **P2** | 次要/邊界，不影響主要功能 | RWD、效能、極端邊界 |

#### 必備的通用模組

不管什麼專案，以下模組和測試都要有：

**認證模組 (AUTH)**
- 註冊、登入、登出
- 權限控制（未登入/普通用戶/管理員）
- Email 等價類（有效/缺@/空白）
- 密碼邊界（太短/太長/空白）

**安全測試 (SEC)**
- 權限：未帶 Token、錯誤 Token、過期 Token
- XSS：所有輸入欄位輸入 `<script>alert(1)</script>`
- SQL Injection：`' OR 1=1 --`
- Path Traversal：`../../etc/passwd`
- CORS：允許/不允許的 Origin

**介面測試 (UI)**
- RWD：桌面 (1920px)、平板 (768px)、手機 (375px)
- 探索性：快速連續點擊、上下頁、多分頁、重新整理

**效能測試 (PERF)**
- API 回應時間
- 連續請求壓力測試
- 大量資料分頁效能

#### 邊界測試通用規則

所有輸入欄位都要測：

| 類型 | 測試值 |
|------|--------|
| 數字 | 0, -1, 最大值, 小數, 文字 |
| 字串 | 空白, 1 字, 正常, 超長 (1000+字), 特殊字元 `' " < > & \` |
| 檔案 | 0KB, 正常, 剛好上限, 超過上限, 錯誤格式 |
| 下拉/選擇 | 不選, 無效值 |
| 日期 | 過去, 未來, 格式錯誤 |

## 分析專案的步驟

### Step 1：了解架構
- 讀 README、SKILL.md、package.json、build.gradle 等
- 確認技術棧（前端框架、後端框架、DB）

### Step 2：掃描 API
- 後端 Controller 目錄 → 列出所有 endpoint
- SecurityConfig → 分辨 public / 需登入 / 需 admin
- 讀 DTO/Model → 了解欄位和驗證規則

### Step 3：掃描頁面
- 前端 app/ 或 pages/ 目錄 → 列出所有頁面
- 讀 API 串接檔 → 了解前後端怎麼對接

### Step 4：識別模組
- 按功能分模組（認證、商品、訂單、支付...）
- 每個模組確認有哪些操作（CRUD + 特殊邏輯）

### Step 5：生成測試案例
- 每個模組按手法分類產出案例
- P0：核心流程（不能壞的）
- P1：重要功能（影響體驗的）
- P2：邊界和進階（加分項）

### Step 6：計算覆蓋率
- 確保每個 API endpoint 至少被 1 個案例覆蓋
- 確保每個頁面至少被 1 個案例覆蓋
- 確保所有輸入欄位都有邊界測試

## 使用方式

將以下資訊提供給 AI：

```
請讀取 QA_GENERATOR_PROMPT.md，然後根據以下專案資訊生成完整的 QA 測試框架。

## 專案資訊
- 專案名稱：{名稱}
- 專案位置：{路徑或 repo URL}
- 測試站 URL：{前台}、{後台/API}
- 測試帳號：{帳號/密碼}
- 技術棧：{前端}、{後端}、{資料庫}
- 其他說明：{特殊功能、注意事項}

請先分析專案結構和所有 API，然後在 qa/ 目錄下生成所有檔案。
```

AI 會：
1. 分析專案 → 列出模組和 API
2. 生成 `qa/README.md`
3. 生成 `qa/QA_SKILL.md`（含完整 API 清單）
4. 生成 `qa/TEST_CASES.md`（按模組 × 手法展開）
5. 生成 `qa/reports/TEMPLATE.md`
6. 生成 `qa/bugs/TEMPLATE.md`

生成完成後，確認：

- [ ] 每個 API endpoint 至少被 1 個測試案例覆蓋
- [ ] 每個前端頁面至少被 1 個測試案例覆蓋
- [ ] 所有有輸入的地方都有邊界測試
- [ ] 安全測試（XSS、SQL Injection、權限）齊全
- [ ] P0 案例覆蓋完整的核心用戶流程
- [ ] 報告和 Bug 模板可直接複製使用
- [ ] QA_SKILL.md 包含足夠資訊讓新手（AI 或人）直接開始測試
